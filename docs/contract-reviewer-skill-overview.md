# 合同审查 Agent Skill

目的：**让任何模型审合同都能审到及格线以上。**

没有 Skill，强模型也能审，但换个小模型就漏一大片；有 Skill，弱模型也被 references 拉到 80%+ 召回，且输出格式、定级标准、修改建议全部结构化。

---

## 一、内容——Skill 怎么写

### 为什么不是"写个好 prompt 就行"

| | 写个 prompt | 用本 Skill |
|---|-----------|-----------|
| 强模型（Claude/Opus） | 审查质量差不多，但格式随 prompt 措辞变 | 格式固定、维度强制覆盖、gotcha 兜底 |
| 弱模型（GLM-4-Flash/Qwen-Turbo） | 召回可能 50-60%，遗漏严重 | references 规则库把弱模型拉到 80%+ |
| 换一个人写 prompt | 结果完全不同 | 同一套 Skill，输出一致 |
| 部署到不同平台 | 每次重写 | 一套 references，Claude Code / LangChain / 纯 API 通用 |

### 三级加载架构

不是一个大 prompt，是一个文件夹 + 按需加载：

```
始终在线（~100 tokens）    → name + description（路由触发器）
激活时加载（~5,000 tokens）→ SKILL.md（审查流程 + gotcha + 自检）
按需读取                   → references/（法条、规则、模板、分级标准）
```

### 文件分工

```
skills/contract-reviewer/
├── SKILL.md                 审查流程 + 10.5 条 gotcha
├── references/
│   ├── purchase-contract.md 12维度规则 + 8个常见风险模式
│   ├── risk-level-guide.md  什么情况判什么级
│   ├── legal-basis.md       法条原文（经国家法律法规数据库验证）
│   └── clause-templates.md  修改模板（复制粘贴入合同）
├── adapters/                 接任何 Agent 框架
└── scripts/                  可观测中间件
```

### 写作原则

"没有这句 Agent 会犯错吗？"——不能通过就删。

- 模型已经知道的常识不写
- 通用规则精简写
- 模型反复犯错的边界（gotcha）详细写——gotcha 是最高价值内容

### 当前的 gotcha

| 会犯什么错 | 怎么防 |
|-----------|-------|
| 条款对一方不利就标风险 | 先确认用户角色 |
| 说"建议完善"不给替代条款 | 每条建议必须可复制粘贴 |
| 不检查检验期合理性 | 强制匹配货物类型 |
| 同时约定诉讼+仲裁没发现 | 立即标严重 |
| 免责条款表面排除间接损失，括号里偷塞直接损失 | 重读完整范围 |
| 违约金0%直接标严重 | 先查其他条款判断笔误 vs 系统设计 |
| 交货时间"甲方通知"但合同有截止日期 | 有锚定=一般，无锚定=重要 |
| 合同空白一律标严重 | 先分三类：不可执行/信息性/无风险 |
| 条款句末悬空一律标严重 | 区分：核心权利不可执行=严重，非核心后果=一般 |

## 二、评估——怎么测

一条命令：

```bash
uv run python evals/run_pipeline.py
```

自动跑 5 步：格式门禁 → 语义匹配 → 质量指标 → 回归对比 → 运营 KPI

### 当前成绩（iter-5，2 份合同 34 个 GT 项）

| | 有 Skill | 无 Skill（baseline） |
|---|---------|-------------------|
| 召回率 | 97.1% | 100% |
| 定级准确率 | ~94%（修正后） | 94.1% |
| 严重风险召回 | 100% | 100% |
| 重要风险召回 | 100% | 100% |

在强模型上，Skill 的增量不在"找得更多"，而在结构化和一致性。Skill 的核心价值在弱模型保底——这部分还没测。

### 护栏

| 守什么 | 怎么守 |
|-------|-------|
| 格式 | 输出必须含总览表+五字段+免责声明，否则不通过 |
| 法条 | 引用的法条必须真实存在，否则标黄 |
| 条款号 | 引用的条款号不能超出合同实际条数 |
| 越界 | 审查不得越界到税务/诉讼策略 |
| 输入 | 非买卖合同优雅退出，不硬套流程 |

### Benchmark 现状

目前是自建评估流程，不是正式 benchmark——2 份合同、单人标注、LLM 裁判未校准。可复现性和可比较性不足，但足以支撑迭代间的相对比较。正式 benchmark 的规格见 `docs/benchmark-spec.md`。

## 三、观测——怎么看

### Eval Pipeline 输出

每次 eval 生成结构化报告（`pipeline_report.json`），包含：

- 门禁结果（格式/法条落地/条款号）
- 语义匹配详情（逐项命中/未命中）
- 质量指标（召回/定级/标高标低/分层）
- 回归对比（与上一轮差异）
- 运营 KPI（token 消耗/审查时长/输出长度）

### 可观测中间件

`scripts/` 下有 telemetry 和 openai-observer，用于在生产环境采集审查 trace。当前处于调试阶段，尚未有真实 trace 数据。

### 迭代历史

| 轮次 | 改了什么 | 召回 | 定级 |
|------|---------|------|------|
| 1-3 | 初始 + gotcha 1-7 | — | — |
| 4 | +gotcha 8（空白分类） | 79.4% | 81.5% |
| 5 | +gotcha 6.5/7.5/8细化 + references 修正 | 97.1% | ~94% |

## 四、韧性——怎么改、怎么扛

### 改完跑 eval，看数字有没有变好、有没有变差

四个杠杆，改哪里取决于问题出在哪：

| 问题 | 动哪里 | 例子 |
|------|-------|------|
| Skill 不该触发时触发了 | description | 商业计划书也触发 → 收紧限定词 |
| Skill 该触发没触发 | description | "审一下这个采购单"没触发 → 加关键词 |
| 漏判某个维度 | SKILL.md | 没查验收条款 → 强化流程步骤 |
| 反复在特定模式犯错 | gotcha | 违约金0%总标严重 → 加"先查其他条款" |
| 定级标准不一致 | reference | 什么时候判严重什么时候判重要模糊 → 补 risk-level-guide |

### 飞轮

```
发现问题 → 分析根因 → 动杠杆 → 跑 eval → 确认修复+无回归 → 下一轮
```

### 已知的薄弱点

| 优先级 | 缺什么 | 为什么重要 |
|--------|-------|----------|
| 🔴 | 弱模型测试 | Skill 的核心价值在保底，不测弱模型等于没验证核心价值 |
| 🔴 | F2 综合分 | 召回+精确的加权综合，一行代码的事 |
| 🟡 | 更多测试合同 | 2 份不能推广，至少 5 份 |
| 🟡 | LLM 裁判校准 | 召回 97% 这个数可能是裁判偏松的结果 |
| ⚪ | 双人标注 GT | 当前单标注者，GT 本身可能有错 |
| ⚪ | 精确率独立计算 | 只知道漏了多少，不知道多报了多少 |
